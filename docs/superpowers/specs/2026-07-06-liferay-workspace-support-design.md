# Liferay Workspace support

## Problem

`liferay-portal-cli` only works against the `liferay-portal` monorepo. Every path-resolution
function hardcodes monorepo assumptions:

- Root detection requires `build.xml` + a top-level `modules/` dir (`internal/portal/root.go`).
- Bundle location comes from `app.server.properties` / `app.server.<user>.properties`
  (`internal/portal/root.go`), which don't exist outside the monorepo.
- Module discovery walks a fixed list of monorepo-only roots (`modules/apps`, `modules/core`,
  `modules/sdk`, etc. — `internal/portal/modules.go`).
- No-arg `build`/`clean`/`sf` shell out to `ant`.
- `client-extension` looks only for the monorepo's internal nested convention,
  `workspaces/<ws>/client-extensions/<name>/client-extension.yaml`.
- `worktree add`'s generic file-propagation step is gated behind `IsPortalRepo`, even though
  most of what it does (symlinking `CLAUDE.md`/`.claude/`, copying `build.*.properties`-style
  globs) has nothing to do with the monorepo specifically.

A **Liferay Workspace** — the standalone client-extension/customer project layout generated by
`blade init` / the `com.liferay.workspace` Gradle plugin, exemplified by `liferay-seo-studio` — has
none of these markers. It has no Ant, a flat `modules/` directory, and its bundle/module locations
are configured through `gradle.properties`.

Goal: `liferay` should work the same way against a Liferay Workspace project as it does against
the monorepo, for every command that has a sensible equivalent.

## Reference project

`/Users/dtruong/Projects/liferay/liferay-seo-studio`:

- `settings.gradle` applies the workspace plugin: `apply plugin: "com.liferay.workspace"`.
- `gradle.properties`: `liferay.workspace.modules.dir=modules`, `liferay.workspace.home.dir`
  (unset here, defaults to `bundles`).
- `modules/` is flat — one directory per module (e.g. `site-initializer-seo-issues`, a plain
  OSGi module with `bnd.bnd`).
- `bundles/` (gitignored) is the assembled/downloaded Liferay bundle — same physical Tomcat/OSGi
  shape as any other Liferay bundle, just living inside the repo instead of alongside it.
- No `client-extensions/` directory in this particular repo, but it's part of the general
  Workspace convention (a top-level sibling of `modules/`).

## Non-goals

- Docker-based deploy (`gw createDockerContainer` / `buildDockerImage` from `Dockerfile.ext`).
  Server/DB lifecycle reuses the CLI's existing native Tomcat + Docker-DB-stack approach, pointed
  at the Workspace's bundle directory, for one consistent mental model across project types.
- Per-environment `configs/<env>` merging beyond whatever `gw initBundle`/`gw deploy` already do.
- Themes (`themes/` dir) — no theme-specific CLI command exists for the monorepo either.
- A diff-scoped source formatter for Workspaces (none exists upstream).

## Design

### Project type detection

`internal/portal/root.go`'s `FindRoot()` currently returns a bare path. It becomes:

```go
type ProjectType int

const (
    Monorepo ProjectType = iota
    Workspace
)

type Root struct {
    Path string
    Type ProjectType
}
```

Per-directory check while walking up from cwd:

- **Monorepo**: `build.xml` + `modules/` both present (unchanged from today's `isPortalRoot`).
- **Workspace**: `build.gradle` or `settings.gradle` exists at that directory and its contents
  match `apply plugin: "com.liferay.workspace"` (Groovy DSL) or `id("com.liferay.workspace")`
  (Kotlin DSL).

Every call site that takes a root path today takes `Root` instead — a mechanical signature change,
no behavior change for existing Monorepo call sites.

### Bundle directory resolution

Branches on `Root.Type`:

- **Monorepo**: unchanged — `app.server.properties` / `app.server.<user>.properties`,
  `app.server.parent.dir`.
- **Workspace**: read `liferay.workspace.home.dir` from the root's `gradle.properties` (default
  `bundles`), resolved relative to the workspace root. No per-user override file: the bundle
  already lives inside the repo, so it's inherently per-checkout / per-worktree.

### Module discovery

`internal/portal/modules.go` branches on `Root.Type`:

- **Monorepo**: unchanged fixed multi-root walk.
- **Workspace**: read `liferay.workspace.modules.dir` from `gradle.properties` (default
  `modules`), walk one level deep for `bnd.bnd`. Additionally check
  `<root>/client-extensions/*/client-extension.yaml` if that directory exists.

Once these two resolvers are type-aware, every command that already just "resolve a module
directory, then shell out to `gw`" — `build <module>`, `sf <module>`, `test`, `test-integration`,
`build-service`, `build-rest` — works unmodified against a Workspace. Same for `server`/`db`
start/stop/restart/status/logs/wipe and the slot/port-offset bundle-patching logic: they only
depend on "where's the bundle," not which project type produced it, since a Workspace bundle has
the same physical Tomcat/OSGi layout as a monorepo one.

### Command-by-command behavior

| Command | Monorepo (unchanged) | Workspace (new) |
|---|---|---|
| `build` (no-arg) | `ant all` | `gw initBundle` if the bundle is missing, then `gw deploy` for every discovered module |
| `build <module>` | `gw deploy -a` | same, generic |
| `clean` (no-arg) | `ant clean` | reject: `"clean requires a module name in a Workspace project"` |
| `sf` (no-arg) | `ant format-source-current-branch` | reject: `"sf requires a module name in a Workspace project — no diff-scoped formatter exists for Workspaces"` |
| `sf <module>` / `test` / `test-integration` / `build-service` / `build-rest` | unchanged | generic, works once module resolution is type-aware |
| `server`/`db` (all subcommands) | unchanged | unchanged — same bundle shape, different resolved path |
| `worktree add` | unchanged | drop the `IsPortalRepo` gate around the generic propagation step (symlinks: `CLAUDE.md`, `.claude/`, etc.; glob copies: `build.*.properties`-style patterns). Skip the Monorepo-only `app.server.<user>.properties` generation. No Workspace-specific file needs generating — the bundle lives inside the repo, so each worktree gets an independent one automatically. Slot/port-offset patching is reused unchanged. |
| `worktree list/remove/prune`, `hosts *` | unchanged | unchanged — already project-agnostic |
| `client-extension` | unchanged (`workspaces/<ws>/client-extensions/<name>/`) | also check `<root>/client-extensions/<name>/client-extension.yaml` |

Rejections (`clean`, `sf` no-arg) surface the specific reason inline rather than a generic
"unsupported" message, so it reads as a deliberate scope decision rather than a bug.

## Testing

**Unit tests** (fixtures under `internal/portal/testdata/`, matching existing test conventions):

- `DetectProjectType`: Monorepo fixture, Workspace fixture (both `build.gradle`- and
  `settings.gradle`-declared plugin forms), neither (plain git repo → error).
- Bundle dir resolution: Workspace fixture with and without a `liferay.workspace.home.dir`
  override in `gradle.properties`.
- Module discovery: Workspace fixture with a flat `modules/` dir plus a `client-extensions/` dir.
- `worktree add`: regression test that generic propagation now runs for a Workspace repo (it was
  previously skipped by the `IsPortalRepo` gate).

**Manual smoke test** against `liferay-seo-studio`:

1. `liferay build` on a fresh clone (no bundle yet) → assembles the bundle and deploys every
   module.
2. `liferay server start` → boots successfully.
3. `liferay build site-initializer-seo-issues` → deploys just that module.
4. `liferay sf site-initializer-seo-issues` → formats just that module.
5. `liferay worktree add` a second worktree of the same repo, confirm it gets its own bundle and
   can run its own server on a separate slot alongside the first.

## Rollout

Single change, no flag — `liferay` detects project type transparently. Monorepo behavior is
unchanged byte-for-byte, so there's no migration step for existing usage.
